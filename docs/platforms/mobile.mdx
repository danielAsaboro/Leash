<Warning>
The mobile app is in early access. On **iPhone** it does on-device chat **and** is a real member of your private mesh (it syncs the task CRDT peer-to-peer). The full tool suite, memory, and computer-use still require the desktop app. Android shares the codebase but mesh membership there is not yet wired (see below).
</Warning>

## What works now

- **On-device chat** — runs fully on-device, no network required after first download; streamed output
- **Private-mesh membership (iPhone)** — pair by scanning an invite QR; the phone runs a real Corestore + Autobase + Hyperbee + Hyperswarm + blind-pairing stack inside a Bare worklet and replicates the same encrypted **task** CRDT as your Macs — not a thin delegated client
- **Borrow a brain** — delegated inference: offload chat to a stronger provider device over the encrypted mesh

## What's coming

- **Remote Leash mode** — the rest of the desktop feature set (memory, tools, larger models) over the mesh, beyond task sync
- **Voice** — on-device STT + TTS
- **Android mesh parity** — the mesh worklet is built for iOS today; Android currently runs chat only

## Requirements

- **Physical device required** — the iOS/Android simulator does not support the native llama bindings. You must run on a real iPhone or Android phone.
- **Expo Go or a dev build** — install via `npx expo start` and scan the QR code with the Expo Go app. For a production build, use `eas build`.
- **Storage** — `Llama 3.2 1B Q4_0` is ~800 MB. Ensure at least 1.5 GB free.

## Running the mobile app

```bash
cd apps/mobile
npm install
npx expo start
```

Scan the QR code with Expo Go on your device. On first launch the model downloads automatically.

## Model details

| Property | Value |

## Why the iOS app runs JSC, not Hermes

The iOS app is pinned to the **JSC** JavaScript engine (`app.json` `"jsEngine": "jsc"` and
`ios/Podfile.properties.json` `"expo.jsEngine": "jsc"`), not Hermes — and that is deliberate, not
an oversight.

The reason is the runtime. `@qvac/sdk`'s React Native code path bundles polyfills (`bare-process`,
`URL`/`URLSearchParams`, web-streams) that ship modern JavaScript syntax Hermes 0.81.5 cannot
compile — running the pods' own `hermesc` over the production bundle produces hard errors like
*"private properties are not supported"*. On Hermes the app red-screens at launch. JSC is a full
engine with no bytecode-precompile limits, so it runs that syntax natively with no Babel workarounds.

The practical rule: **don't "fix" the engine back to Hermes.** If a future SDK release claims Hermes
support, re-verify before switching — compile the `expo export:embed` bundle with the pods'
`hermesc` first, and only flip if it compiles clean. The build recipe lives in
[iOS install](/install/ios).
