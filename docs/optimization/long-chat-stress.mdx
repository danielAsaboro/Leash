The long-chat stress runner exercises the product path users actually use: repeated `POST /api/leash/chat` calls with normal chat messages. It mixes quick turns, tool turns, broker turns, health safety, file search, and continuity markers so the route has to survive a long conversation instead of one idealized request.

## Scenarios covered

| Scenario | What it exercises |
| --- | --- |
| Quick direct | No-model direct chat path. |
| No-tool-word guard | Ensures lightweight prompts that mention tools do not force the full agent. |
| Files date | Registered `bash` through chat and goal-run evidence. |
| File-finder metadata | Skill metadata answer without disk search. |
| File-finder search and no-result | Bounded read-only local search. |
| Context broker | `context_run` graph search. |
| Memory broker | Memory recall/read broker. |
| Tasks broker | Task list/read broker. |
| Daily paper broker | `understory_today` broker action. |
| Health route | Read-only health context with non-diagnostic answer. |
| Continuity turns | Long-tail chat after previous tool calls. |

## Acceptance bar

A run is only useful if it proves the chat surface held together. The runner checks HTTP success, `text/event-stream`, conductor metadata, observed routes, observed skills, and observed tools. Internal helper scripts can isolate failures, but the long stress run is the product-path proof.
