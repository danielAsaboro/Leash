Mycelium is a private, offline-capable exocortex that runs across your own device mesh.
It keeps inference, embeddings, retrieval, delegated compute, and memory loops on your
hardware through `@qvac/sdk`. The headline surface is **Leash**, the assistant and
dashboard in `apps/web`, but Leash is only one part of the platform.

## Surfaces and components

- **Mycelium** is the whole platform: the local-first runtime, the mesh, the memory loop,
  and the workspace that builds them.
- **Leash** is the flagship interface: the assistant and operator console in `apps/web`.
- **Brain** is Leash's configuration surface for memory, tools, prompts, models, MCP,
  growth, and operator controls.
- **The agent harness** is the extensibility story: tools, MCP servers, skills,
  computer use, connectors, and local daemons can be composed to pursue arbitrary work.
- **Feed** is The Understory surface: the autonomous paper generated about your world.
- **Services** is the daemon console: the place where background processes are started,
  stopped, inspected, and restarted.
- **Hypha** is the delegated-compute and pairing daemon in `apps/hypha`.
- **Hub** and **Edge Node** are narrower runtime apps used to exercise the original mesh
  and replicated-graph paths.
- **Newsroom**, **Broker**, **Watch**, **Cron**, and **MCP** are supporting runtimes.

## Non-negotiable constraints

- **QVAC only.** All inference, embeddings, RAG, delegated compute, and LoRA work go
  through `@qvac/sdk`. There is no cloud AI fallback.
- **Offline after warm-cache.** First run can download model weights and bootstrap the
  DHT. After that, local chat, retrieval, and mesh operations are expected to work
  without network access.
- **Real runtime code only.** The workspace adds packages and apps only when they land as
  working implementations. The spike, smoke tests, and daemons are all real paths.
- **Audit logs stay on.** Spike and runtime flows emit JSONL audit records and are part
  of the verification story.

## Docs conventions

These docs follow Diataxis on purpose:

- Tutorials teach one happy path.
- How-to guides solve one concrete job.
- Reference pages stay factual and lookup-oriented.
- Explanation pages handle architecture, rationale, and tradeoffs.

Hackathon evidence, spike results, and build logs stay in `submission/`, not in this
docs tree. When a docs page needs to point at evidence, it links out instead of copying
the material.

## What you can make it do

These are the kinds of jobs the current system is designed to take on:

- **Ground and answer private questions** by combining chat, local retrieval, typed
  memory, and notes under `data/notes/`.
- **Operate your machine with approval** by combining computer-use tools, file access,
  shell execution, and GUI control.
- **Borrow another device's compute** by combining the broker, Hypha, and a warm peer.
- **Add new integrations without rewriting the app** by attaching MCP servers and
  letting their tools merge into the assistant's registry.
- **Teach repeatable procedures** by importing or writing skills with instructions and,
  when needed, script-backed steps.
- **Run long-lived background behavior** through daemons like watcher, newsroom, cron,
  and MCP.
- **Publish a local paper** by combining research, review, generation, and feed surfaces
  in The Understory.

The important point is not the individual features. It is that these pieces compose.
