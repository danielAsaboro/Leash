title: "How to use Brain and Settings"
description: "Operate the configuration surfaces that control memory, tools, prompts, models, secrets, and mesh settings."
---

Use this guide when Leash is already running and you want to configure how the system
behaves, what it knows, and which integrations and devices it can use.

## Use Brain as the operator surface

Open `/brain`. This is where you tune the assistant itself.

The main tabs are:

- **Memory** for recent saved memories, note and activity counts, and links into the
  full memory and notes browsers
- **Skills** for enabling, disabling, importing, and editing skill bundles
- **Tools** for availability and ask-first control
- **MCP** for connected server integrations, including the built-in mesh tools daemon
- **Prompts** for system, voice, and specialist prompt overrides
- **Models** for the served model catalog, download state, fit signals, and sharing
- **Growth** for personal adapter progress and eval deltas
- **Forage** for the latest background discovery results

## Tune memory and tool behavior

Use **Brain → Memory** when you need to inspect what the assistant has already saved.
Use **Brain → Tools** when you need to:

- disable a tool completely
- require approval before a side-effecting tool runs
- verify which tool surfaces are actually available to the assistant

If chat starts hallucinating actions it can no longer take, this is the first place to
check.

## Manage skills and MCP integrations

Use **Brain → Skills** to review and enable imported skills deliberately. Imports land
disabled by default so you can inspect them first.

Use **Brain → MCP** to add external MCP servers by URL or manage the built-in mesh
tools server. MCP servers extend the assistant's tool surface and should be treated as
runtime integrations, not as static docs examples.

## Control prompts and models

Use **Brain → Prompts** when you need to adjust assistant behavior without changing
code. Leave a prompt blank to fall back to the shipped default.

Use **Brain → Models** when you need to:

- download a model through the QVAC network
- inspect whether it fits the machine
- check whether it is loaded, cached, or shared to peers
- start from the model inventory rather than from the serve config file

## Use Settings for machine-local state

Open `/settings` when you need to manage the machine rather than the assistant prompt.

The main tabs are:

- **Storage** for model cache and app-data cleanup
- **Devices** for mesh membership, invites, PIN pairing, disconnect, and reset
- **Secrets** for Home Assistant and SearXNG credentials
- **Permissions** for microphone, screen recording, and accessibility
- **About** for build and license identity

## Use the right surface for the job

- If you are changing assistant behavior, start in **Brain**.
- If you are changing local machine state, start in **Settings**.
- If you are operating processes, use **Services** instead.
