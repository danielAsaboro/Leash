---
title: "Borrow a modality from a peer"
description: "Advertise vision, embeddings, speech-to-text, or TTS on one device and borrow it from another over the mesh."
---

When a device lacks a model — no vision, no embeddings, no speech — it can borrow that
capability from a paired peer that serves it. This guide advertises a modality on a provider and
borrows it from a consumer. It assumes a [paired mesh](/guides/pair-a-second-device). For why
non-chat modalities take a different path than chat delegation, see
[Modality borrowing](/explanation/modality-borrowing).

## 1. Advertise the modality on the provider

On the device that **has** the model, two things must be true:

1. The model is configured and served (e.g. `qwen3vl` for vision, `gte-large` for embeddings,
   `parakeet` for STT, `supertonic` for TTS).
2. It is **shared** to the mesh — flip the model's **Share** toggle in
   [Models](/guides/use-the-model-catalog), or leave the deny-set empty to share everything.

A peer only advertises aliases it can actually serve locally, so a shared alias whose weight is
missing won't appear.

## 2. Enable the forward path on the consumer

Borrowed modalities travel a **forward path** that is off by default. On the **borrowing**
device, enable it in the Hypha environment:

```bash
HYPHA_FORWARD=1
```

This is what makes the consumer willing to proxy vision, embeddings, STT, and TTS to a peer.
(Chat delegation works without it — it rides the SDK's own peer call.)

## 3. Borrow

Use the modality normally on the consumer — ask an image question, run a retrieval-heavy turn,
dictate, or play a message aloud. With no local model for that modality but a sharing peer
present, the request is forwarded to the peer, run against its serve, and streamed back. From
your seat it looks like one assistant; the work happened on the peer's GPU.

## What failover does

If more than one peer serves the alias, the forward path tries them lowest-tier first and fails
over to the next on any error — but only **before** the answer starts streaming. Once the first
byte reaches you, a mid-stream failure surfaces as an honest error rather than a silent retry
(switching peers would corrupt the stream).

If settlement is on and the chosen peer advertises a paid rail, the request binds to that one
peer for the billed session — no failover mid-settlement. See
[Enable the agent economy](/guides/enable-the-agent-economy).

## Verify

- The borrowed answer appears on the consumer even though it serves no model for that modality.
- The provider's `apps/hypha/logs/hypha.jsonl` records the forwarded request with its billable
  usage stamp.
- **Settings → Devices** on the consumer lists the peer's advertised models for that modality
  (● warm / ○ cold).
