---
title: "Computer use"
description: "Let the assistant read files, run commands, and drive the GUI — with a permission slip."
---

Leash can act on the Mac it runs on. Six tools cover seeing the screen, files, shell,
and mouse/keyboard — all driven by a **local or mesh-delegated** model, never a cloud
API. This guide shows how to use them and how to widen or narrow what they're allowed
to touch.

<Warning>
`run_command` executes real shell commands as your user, and `computer` clicks and types
wherever you approve. The approval card is the boundary — read it before you approve.
</Warning>

## Ask for file work

> read ~/.zshrc and tell me what the PATH export does

`read_file` runs without approval (read-only, jailed to your home directory). Writes and
edits pause for approval:

> create a file ~/notes/draft.txt saying "hello"
> in ~/notes/draft.txt, change hello to goodbye

`edit_file` is exact-string replacement — if the text it targets isn't unique in the
file, it refuses and says so rather than guessing.

## Run a shell command

> run `ls -la ~/Downloads` and tell me what's biggest

You'll get an approval card with the exact command. Output is capped at 16 KB and the
command is killed after 60 seconds. The working directory defaults to your home folder.

## See the screen

> take a screenshot — what app is focused?

The frame is captured, described by the local vision model, and the PNG is deleted
immediately. The terminal running the dashboard needs **Screen Recording** permission
(System Settings → Privacy & Security).

<Note>
On the current serve build, vision inference has a known upstream regression — the tool
reports the failure honestly in-chat. See [Troubleshooting](/troubleshooting).
</Note>

## Drive the mouse and keyboard

Requires [`cliclick`](https://github.com/BlueM/cliclick) (`brew install cliclick`) plus
**Accessibility** permission. Then:

> click at 200,300 · type "hello" · press return · scroll down 2 pages

Coordinates are logical screen points — on a Retina display, divide pixel coordinates
from a screenshot by ~2. This tool is experimental: grounding accuracy scales with the
driving model's size (see below).

## Use a bigger driver — optionally over the mesh

The default driver is the local chat model. For stronger GUI control, point computer
turns at a bigger served alias:

```bash
# locally (needs the RAM):
LEASH_COMPUTER_MODEL=gpt-oss-20b npm run web:dev

# delegated — a paired peer serves it warm, the broker routes the turn over the mesh:
LEASH_COMPUTER_MODEL=gpt-oss-20b QVAC_OPENAI_URL=http://127.0.0.1:11436/v1 npm run web:dev
```

**Brain → Tools** shows which model drives the computer tools and whether it runs
locally or on a named peer.

## Narrow or widen the jail

| Knob | Default | Effect |
|---|---|---|
| `LEASH_COMPUTER_ROOT` | your home directory | hard jail for read/write/edit (symlink-escape proof) |
| `LEASH_COMMAND_ALLOW` | unset | comma-separated first tokens `run_command` may use — a guard-rail, **not** a security boundary |
| Brain → Tools toggles | screenshot + read_file un-gated | flip "Ask first" per tool, or disable a tool entirely |
