Use this guide when the core assistant is not enough and you want to teach it new
procedures, add new integrations, or widen what work it can accomplish.

## Think in harness terms

Leash is not just a chat UI with a fixed tool list. The shipped runtime already lets
you combine:

- built-in tools for retrieval, memory, Home Assistant, image generation, activity, and time
- computer-use tools for files, shell, screenshots, and GUI control
- MCP servers that contribute new tool namespaces
- skills that add instruction bundles and optional executable scripts
- background daemons such as Hypha, watcher, newsroom, cron, and MCP itself

That combination is the harness. You add capabilities, then the assistant can route work
through them.

## Add or remove built-in tools

Open **Brain → Tools** when you want to change the built-in tool surface.

Use it to:

- disable tools that should never be available
- require approval before risky actions
- inspect which capabilities are currently exposed to the model

This is the fastest way to shape what the assistant is allowed to do.

## Add an MCP server

Open **Brain → MCP** and connect a server by URL, or use the built-in mesh tools server.

MCP is the main way to bolt on a new tool family without changing the core assistant
code. If a server exposes tools and Mycelium connects to it successfully, those tools
join the assistant's registry and can participate in normal tool-calling.

This is how you expand into external systems while still keeping the main runtime local.

## Add a skill

Open **Brain → Skills** to create or import a skill.

Use skills when you want to teach the assistant:

- a reusable procedure
- a house style or workflow
- a domain-specific decision pattern
- a small script-backed capability that should not become a first-class built-in tool

Skills are especially useful when the missing behavior is mostly procedural rather than a
new API integration.

## Use computer tools when the target is the machine itself

The built-in computer tools already let the agent:

- read and edit files
- run shell commands
- inspect the current screen
- click, type, and navigate the GUI

That means many "I want the agent to do X" requests do not require a new integration at
all. They require the existing harness plus the right approvals and model route.

## Use services when the capability needs a daemon

Some capabilities are not just tools. They are long-lived processes:

- Hypha for mesh routing and delegation
- watcher for ambient activity sensing
- newsroom for the paper
- cron for recurring work
- MCP for in-chat integration flows

In those cases the harness is wider than the tool registry. It includes the runtime
processes that keep the capability alive between turns.

## Choose the right extension path

- Use **Tools** when the capability already exists and you just need to allow, deny, or gate it.
- Use **MCP** when you need a new external integration surface.
- Use **Skills** when you need a reusable procedure or script-backed workflow.
- Use **Computer use** when the target is the local machine itself.
- Use **Services** when the capability needs a background daemon.

## What this lets you do

Once those pieces are composed, Mycelium stops being "the chat app that ships with some
features" and becomes "the local agent harness I can keep teaching." That is the
important claim.

Concrete examples:

- attach an MCP server, then let the assistant use those tools inside normal chat turns
- import a skill, then have the assistant follow your house workflow every time that job appears
- use computer tools plus approvals to inspect files, run commands, and drive the machine
- run Hypha plus the broker so heavier work can shift to another one of your devices
- combine watcher, research, and newsroom so the system keeps producing work outside chat

For full workflow composition patterns, use
[How to compose agent workflows](/guides/compose-agent-workflows).
