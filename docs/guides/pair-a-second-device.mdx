---
title: "How to pair a second device"
description: "Connect a second machine to the encrypted Mycelium mesh with Hypha."
---

Use this guide when you already have one machine running and want to add another device
to the mesh.

## Assumptions

- both machines have the same `mycelium/` code
- both machines have dependencies installed
- both machines can reach each other on the same LAN
- both machines can run `npm run hypha`

## Start Hypha on both machines

On each machine:

```bash
npm run hypha
```

That starts the local Hypha control plane on `:11437`. Pairing traffic uses the LAN
pairing port `:11438`.

## Pair with the CLI path

On the machine that will host the invite:

```bash
npx tsx apps/hypha/src/main.ts invite
```

Copy the invite code it prints.

On the joining machine:

```bash
npx tsx apps/hypha/src/main.ts pair <invite-hex>
```

When the join succeeds, start the normal daemon on both machines again:

```bash
npm run hypha
```

## Pair through Leash and MCP

If the web app and `npm run mcp` are running, you can also pair through the assistant.
The bundled MCP server exposes mesh pairing tools and asks for the PIN or device choice
as an in-chat form.

## Pair programmatically (no PIN)

For a **fresh** device, the invite *is* the capability — you can pair over Hypha's HTTP
control routes without a PIN. Mint an invite on the host for a named mesh:

```bash
curl -s -X POST http://127.0.0.1:11437/mesh/invite \
  -H 'content-type: application/json' -d '{"meshId":"primary"}'
# → {"ok":true,"invite":"<invite>"}
```

Then join on the new device with that invite:

```bash
curl -s -X POST http://127.0.0.1:11437/mesh/join \
  -H 'content-type: application/json' -d '{"invite":"<invite>"}'
# → {"ok":true,"meshId":"<id>"}
```

<Note>
This works for a device that has never been a member. Re-promoting an existing-but-broken
member (one that was disconnected or had its mesh store wiped) needs the PIN/MCP flow above —
the programmatic join hangs on an already-known identity.
</Note>

To drop a joined mesh later, `POST /mesh/leave {"meshId":"<id>"}` (any member; the primary
mesh can't be left). See [The mesh and membership](/explanation/mesh-and-membership) for the
visibility, founding-writer, and ghost-cleanup model behind these routes.

## Verify the pair

Check Hypha health on the local machine:

```bash
curl -s http://127.0.0.1:11437/health
```

Expected result: the peer count should rise once the other device is live and reachable.

If you are running the broker, also inspect:

```bash
curl -s http://127.0.0.1:11436/__broker/stats
```

When a peer advertises warm aliases, broker overflow can route there.

## If pairing does not show up

- make sure both Hypha daemons are running, not just the CLI invite/pair helper
- confirm both devices are on the same LAN
- confirm the joining device uses the exact invite code
- confirm the peer has the relevant model file locally if you expect that alias to be advertised
