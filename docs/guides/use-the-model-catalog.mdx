title: "Use the model catalog"
description: "Find a model, download it, share it to the mesh, and tune its compute and context size to fit your machine."

The **Models** tab (under Brain) is one filterable browser over the full QVAC catalog plus the
models you've already configured. This guide covers finding and downloading a model, sharing it
to your mesh, and tuning how it runs. For the concepts behind aliases, compute, and context
size, see [Models and the catalog](/explanation/models-and-the-catalog).

## Find a model

Open **Brain → Models**. Narrow the list with the two facet rows:

- **Status** — All / Configured / Downloaded / Available.
- **Kind** — All / Text / Image / Speech / Embedding / OCR / Translation.

"Available" is the catalog residual — 700+ models you could run but haven't downloaded. (The
`gpt` and `bitnet` families are hidden from this view unless you've already configured one.)
"Configured" is the short list your serve actually exposes as aliases.

## Download a model

Pick an available model and start the download. It runs in a detached child process, so it
survives a dev-server restart, and the row shows live progress. Downloaded weights land under
`~/.qvac/models`.

<Note>
If your serve had to open the registry to fetch blobs, a concurrent dashboard download can fail
with a file-descriptor lock error — the registry corestore is single-process. Stop Model Serve
and Hypha, download, then start them again.
</Note>

## Configure it as an alias

Adding a downloaded model to the serve config gives it a short **alias** and sets `preload`
on by default so it warms at startup. Routes target the alias, so this is the step that makes
the model usable by chat, retrieval, or a borrow.

## Tune compute and context size

Each configured model has two controls in its row:

- **CPU / GPU** — pin the model to CPU, GPU, or the SDK default (GPU on Apple Silicon). Move a
  low-priority model to CPU to free the GPU for the chat model.
- **Context size** — drag the slider (≈1k–32k tokens). The fit estimate updates live and flags
  the result as **fits**, **tight**, or **too big** for your RAM. Larger context means a larger
  KV cache, so size it to the conversations you actually have.

Both changes apply the next time the serve loads the model, not instantly — the serve has no
live-reload for a resident model.

## Share a model to the mesh

Flip the **Share** toggle on a model's row to advertise it to mesh peers. Sharing is per-model:
shared aliases appear in peers' catalogs and become borrowable; unshared ones stay private via a
daemon deny-set. This is the prerequisite for
[borrowing a modality](/guides/borrow-a-modality-from-a-peer) — a peer can only borrow what you
share.

## Verify

- A downloaded model shows under **Downloaded**; a configured one under **Configured**.
- `curl http://127.0.0.1:11435/v1/models` lists every served alias.
- A shared model appears in a paired peer's **Devices** view under your device's advertised
  models (● warm / ○ cold).
