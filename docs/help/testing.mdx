## Fast health check

The fastest low-friction checks in the current repo:

```bash
# 1. Is the serve responding?
curl http://127.0.0.1:11435/v1/models | jq '.data[].id'

# 2. Is the web server up? (the public landing route returns 200)
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:6801/

# 3. Do the deterministic fast paths still pass?
npm run smoke:direct-answer
```

For the real authenticated chat path, use:

```bash
LEASH_COOKIE='leash_session=...' npm run stress:long-chat-turn
```

The stress script also accepts `/tmp/leash-cookie.txt` instead of `LEASH_COOKIE`.

## Script-first verification

These are the current repo-backed checks to reach for first:

| Area | Best current check | Notes |
|---|---|---|
| Direct answers / simple chat heuristics | `npm run smoke:direct-answer` | offline logic smoke |
| File fast path | `npm run smoke:file-finder-fast-path` | bounded search path |
| Brokered context / memory / tasks | `npm run smoke:tool-brokers` | read-only broker tools |
| Health routing and policy | `npm run smoke:psy-health` | offline health-lane policy + specialist wiring |
| Multi-agent orchestration | `npm run smoke:orchestration` | live serve required |
| Plugin activation on real embeddings | `npm run smoke:plugins-live` | live serve required |
| Full authenticated chat surface | `LEASH_COOKIE='...' npm run stress:long-chat-turn` | live web app + valid session cookie |

## Heartbeat verification

`POST /api/leash/heartbeat` is **not** a browser-session route. It is protected by the shared internal token.

```bash
# Trigger one heartbeat cycle manually (read the token from the active data dir)
curl -X POST http://localhost:6801/api/leash/heartbeat \
  -H "x-leash-internal: $(cat <dataDir>/.leash-internal-token)"
```

A healthy response is a JSON heartbeat result. If you are not debugging the server-to-server path specifically, using the Scheduler / Services UI is the safer check.

## Checking audit logs

Audit logs are per-process in the current runtime. Useful starting points:

```bash
jq 'select(.event=="completion") | {modelId, ttftMs, tokensPerSecond}' apps/hypha/logs/hypha.jsonl
jq 'select(.event=="completion") | {modelId, ttftMs, tokensPerSecond}' spike/logs/01-inference.jsonl
```

For the full schema and the current log locations, see [Audit log](/reference/audit-log).

## After an `npm install`

After any `package.json` change or `npm install`, always restart:
1. The web server (`npm run web:dev`)
2. The broker / Hypha if you use them (`npm run broker`, `npm run hypha`)
3. Any standalone MCP daemons you launched (`npm run mcp`, `npm run tools-mcp`)
4. The screen watcher or newsroom if running (`npm run watch`, `npm run newsroom`)

The scheduler is the dashboard-managed `mcp-cron` service, not a root `npm run cron` script.

## After a config change

After any change to `qvac.config.base.json`:
1. Restart `qvac serve openai` — there is no live-reload API
2. Wait 30–60 seconds for models to preload (check `curl .../v1/models` until all aliases show READY)
3. Run the smoke test above
