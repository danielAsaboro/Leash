## The serve is not responding

**Symptom:** Chat shows "serve is down" or requests timeout with no response.

**Fix:**
1. Check that `qvac serve openai` is running: `curl http://127.0.0.1:11435/v1/models`
2. If it's not running, start it: `qvac serve openai`
3. Check `QVAC_OPENAI_URL` — it should point to `:11435` (not the broker at `:11436`)
4. If the serve starts but exits immediately, check that at least one alias is in `qvac.config.base.json` and the weight file is downloaded.

## The chat model hangs — no tokens generated

**Symptom:** The chat spinner runs but no tokens appear; eventually times out.

**Common causes:**
1. **Zero tools on a `tools: true` route** — the `qwen3-4b` serve hangs at zero tokens on toolless requests. This happens if all tool groups are disabled in Brain → MCP. Enable at least one group.
2. **Too many schemas** — if `LEASH_DEBUG_TOOLS=1`, the log shows the active tool count. Above ~22 schemas the prompt overflows. Disable some MCP groups.
3. **Model not preloaded** — the serve loads models lazily on first request. The first call can take 30–60 seconds on slow storage. Wait and retry.
4. **Temperature issue** — the serve default temperature (~0.1) causes repetition and hangs on Qwen3. Leash overrides this per-turn (0.6 or 0.7) — if you're calling the serve directly, set `temperature: 0.7`.

## The screen watcher isn't running

**Symptom:** `active_context` and `activity_recent` return empty; the heartbeat has no activity to read.

**Fix:**
1. Start it: `npm run watch` (or the desktop app starts it automatically)
2. Grant **Accessibility** and **Screen Recording** permissions in System Settings → Privacy & Security
3. The VLM summary requires `qwen3vl` to be served — without it, watcher logs app + window title but no summary

## Voice call isn't transcribing

**Symptom:** VoiceCall enters "transcribing" state but returns empty or no text.

**Fix:**
1. The Parakeet STT rejects `webm/opus` — Leash re-encodes to WAV before sending. If you're seeing empty transcripts check browser console for decode errors.
2. Mic permission denied → state goes to `denied`. Grant mic permission in browser settings.
3. Check that `parakeet` alias is served and READY in Brain → Models.
4. Try a louder utterance — VAD onset threshold is 0.04 RMS. In a noisy environment it may trip too early and clip your speech.

## Memory / RAG search returns empty

**Symptom:** `search_graph` returns no results; grounded answers cite nothing.

**Fix:**
1. Check that `embed` is served — without embeddings, the graph store can't index or retrieve.
2. Check Brain → **Local context** — if the note file is not visible there, the assistant has nothing local to retrieve.
3. There is no root `npm run index` script or in-app re-index button in the current repo. For a lower-level ingest/retrieval proof, run `npm run senses:smoke`.
4. Check `data/notes/` — if there are no `.md` files, there's nothing to index.
5. Check that the Memory group is enabled in Brain → MCP.

## Hypha daemon won't connect to peers

**Symptom:** Mesh shows no peers; `POST /hypha/peers` returns empty.

**Common causes:**
1. **LAN issue** — hyperdht uses a LAN shortcut that fails silently when the LAN probe returns no response. Set `HYPHA_DEBUG_LANPROBE=<host:port>` to diagnose.
2. **Same seed used twice** — each device needs a distinct `QVAC_HYPERSWARM_SEED`. Two processes sharing a seed half-open every dial. Generate a new seed per device.
3. **Orphaned background process** — macOS denies Local Network to processes whose SSH session has exited. Launch hypha session-attached: `ssh -n -f pro 'zsh -lc "cd ~/mycelium && npx tsx apps/hypha/src/main.ts"'`
4. **Registry lock** — if the serve and hypha both try to open the corestore registry, one holds the fd-lock. Stop model serve before downloading new models on a machine where serve has touched the registry.

## Chat won't load — `MODEL_SRC_TYPE_MISMATCH`

**Symptom:** The serve refuses to load your chat alias and the worker throws `MODEL_SRC_TYPE_MISMATCH`.

**Cause:** You pointed the alias at `QWEN3_4B_Q4_K_M` — the constant **without** `INST`. Despite the name, that SKU is a **diffusion** model (`sdcpp-generation` engine), not an LLM, so loading it as chat is rejected.

**Fix:** Use the instruct variant `QWEN3_4B_INST_Q4_K_M` — the real 4B chat LLM (this is what `qvac.config.base.json` ships). Always check a constant's catalog `engine`/`addon` before trusting its name; only the `…_INST_…` Qwen3 4B constant is a chat model. See [Configuration](/models/config) and [Chat & reasoning](/models/modality/chat).

## Tool calls are never made

**Symptom:** The agent answers in prose but never calls a tool, even when a tool is clearly needed; `LEASH_DEBUG_LOOP=1` shows `finish=stop toolCalls=0`.

**Fix:** Tool calls are silently never parsed unless the **serve config** opts in. The chat alias in `qvac.config.base.json` must set:

```json
"config": { "tools": true, "toolsMode": "dynamic", "ctx_size": 32768 }
```

Without `tools: true` + `toolsMode: "dynamic"` the model returns plain text and tool-call syntax is ignored. A too-small `ctx_size` also drops the tool schemas off the front of the prompt — keep chat aliases at the `32768` default (`DEFAULT_CTX_SIZE`). See [Configuration](/models/config). Note the opposite trap (a `tools: true` route with **zero** active tools hangs at zero tokens) under [The chat model hangs](#the-chat-model-hangs-no-tokens-generated).

## Qwen3 overflows context while "thinking"

**Symptom:** A Qwen3 turn fails with `processPromptImpl: context overflow`, or emits a long chain-of-thought and then no answer.

**Fix:** Qwen3 emits a chain-of-thought that overflows a small context. Two things fix it:

1. Set `ctx_size ≥ 2048` (chat aliases already default to `32768`).
2. Disable thinking with `reasoning_budget: 0` — and note it lives in **`generationParams`**, not `modelConfig`:

```ts
completion({ modelId, history, generationParams: { predict, reasoning_budget: 0 } })
```

Leash already passes `reasoning_budget: 0` on its router, critic, and Council turns. See [Configuration](/models/config).

## Citations vary or go missing on a small model

**Symptom:** A grounded answer cites a source on one run and forgets to on the next, on a 1B-class model.

**Cause:** Citation reliability is a **model ceiling** on small models — not a configuration bug. A 1B model simply varies in whether it attaches the citation.

**Fix:** Run the answer through the [Council](/agents/council), which exists to compensate: multiple passes (drafter / verifier / critic) raise citation consistency above what a single small-model turn delivers. Or serve a larger chat alias for grounded routes.

## Model RAM reads as ~0 in the Node process

**Symptom:** Logging `process.memoryUsage().rss` shows almost no delta after loading a multi-GB model, so it looks like nothing loaded.

**Cause:** Models run in a **separate Bare worker process**, not the Node process — so Node's RSS never reflects model memory.

**Fix:** Measure true model RAM in **Activity Monitor** (look for the Bare worker), not via `process.memoryUsage()`. This is expected, not a leak or a failed load.

## The desktop app won't open

**Symptom:** App bounces in the dock then closes, or shows a blank window.

**Fix:**
1. Check that `LEASH_DATA_DIR` or the data folder chosen at setup still exists and is writable.
2. Open Console.app and filter by "Leash" to see crash logs.
3. Delete `~/Library/Application Support/Leash/install.json` to re-run the setup wizard.
