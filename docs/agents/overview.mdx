Leash is a **single, persistent agent**: every turn runs through the same `ToolLoopAgent`, wired to on-device inference through `@qvac/sdk`. What changes per turn is the *route* (which model drives, which tools are active), the *prompt stack* ([constitution, skills, route notes, and overrides](/agents/prompts)), and the *sampling* (thinking vs. non-thinking). This hub maps the architecture.

## The five routes

Every incoming message is classified into one of **five routes** before the agent loop starts. The route picks the driving QVAC model and narrows the default toolset.

| Route | Driving model | Default toolset | When |
|-------|--------------|-----------------|------|
| `chat` | `qwen3-4b` (alias `chat`) | Everything except computer + MCP-admin tools | Default for text turns |
| `health` | `medpsy` (QVAC's Qwen3 fine-tune) | Same as `chat` | Medical / mental-health intent |
| `files` | `qwen3-4b` | Bash lane only (read-only shell) | File-heavy work |
| `computer` | Computer-use model (defaults to chat) | Computer tools only | Screen / GUI turns |
| `vision` | `qwen3vl` | None â€” single-shot, no tool loop | Image / screenshot questions |

See [Routes](/agents/routes) for the full routeâ†’modelâ†’toolsetâ†’approval-posture table.

## Skills-first, constitution-grounded

When an **active skill** declares its own `tools:` set, that set becomes the exact toolset for the turn â€” the skill is the primary lane and route selection is the fallback. Underneath it all, `soul.md` and `goals.md` are folded into **every** turn's system prompt, so the agent is goal-aware in chat, not only in proactive heartbeats.
