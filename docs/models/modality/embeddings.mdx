Embedding models turn text into vectors for semantic search. In the SDK catalog they carry
`endpointCategory: "embedding"`. Leash leans on embeddings heavily — they're the backbone of
grounded chat.

## Models Leash uses

| Alias | SDK model | Size | Powers |
|-------|-----------|------|--------|
| `embed` | `GTE_LARGE_FP16` | ~670 MB | RAG retrieval, on-goal scoring, notification dedup |

## What it powers

`embed` is required for the features that make Leash *grounded*:

| Feature | How embeddings are used |
|---------|------------------------|
| RAG search (`search_graph`, `recall`) | Indexes notes/memories and retrieves passages by similarity |
| Memory indexing | Embeds new notes and memories as they're written |
| Skill activation | Scores the user message against each skill's description embedding |
| Heartbeat dedup | Fuzzy-matches a new proposal against recent ones (cosine ≥ 0.92 = duplicate) |
| On-goal scoring | Measures how relevant a heartbeat proposal is to your `goals.md` |

<Warning>
**Grounded chat requires `embed`.** Without it, `search_graph`/`recall` return nothing, skills stop
auto-activating, and the heartbeat can't dedup — grounded chat silently degrades. It's part of
the minimum viable kit alongside your main chat alias.
</Warning>

## Serve config

Embedding models need no special config block:

```json
"embed": {
  "model": "GTE_LARGE_FP16",
  "preload": true
}
```

## Adding an embedding model

Wire it from the [Assistant Kit](/models/catalog) (recommended — it's one of the four kit roles),
or download an embedding model from **Brain → Models → Catalog** (filter by **Embeddings**) and
configure it under the `embed` alias.

<Note>
If you swap the embedding model, treat it as an index rebuild event. Vectors from a different model
aren't comparable, so a swap without rebuilding retrieval breaks grounding. The current repo does
not expose a one-click re-index button; validate the ingest path with `npm run senses:smoke`
before trusting retrieval again.
</Note>
